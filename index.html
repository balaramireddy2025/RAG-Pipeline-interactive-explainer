STEP 01

What is RAG & Why You Need It

LLMs are great at language, but they don’t automatically know your private documents and they can still confidently hallucinate. RAG fixes both by retrieving the most relevant passages from your knowledge base and grounding the LLM’s response in that context.

🚫

Problem 1: No Private Data

GPT-4 was trained on public internet data. It knows nothing about your company docs, internal PDFs, or proprietary knowledge. Ask it about your HR policy? It will guess—or worse, make something up.

😵

Problem 2: Hallucinations

LLMs confidently generate false information. They don’t “know”—they predict the next token. Without grounding in real data, they fabricate facts, dates, numbers, and quotes.

✅

Solution: RAG

Retrieval-Augmented Generation: first retrieve relevant information from your documents, then feed it to the LLM as context. The answer is grounded in your actual data, not guesses.

💡
Think of it like “open-book QA”. Instead of asking the model to answer from memory, RAG retrieves the exact pages (chunks) from your knowledge base and shows them to the LLM as context. The LLM can then answer with real, traceable evidence.
📐

Two Phases

Indexing (offline): load → preprocess → chunk → embed → store. Querying (online): embed query → retrieve → augment → generate.

🎯

Why Not Fine-Tune?

Fine-tuning is expensive, slow to update, and still hallucinations-prone. RAG is cheaper, always up-to-date, and traceable—you can point to the document passages behind the answer.

⚡

Production Ready

RAG powers enterprise Q&A, support bots, code assistants, legal research, and other grounded knowledge workflows—where “answer quality + citations” matters.

STEP 02

Document Loading

Ingest raw source material into a uniform text format. PDFs, Word docs, HTML, CSVs, and code all become a Document object with page_content (text) and metadata (e.g., source filename + page number). This “metadata-first” design is what makes citations and filtered retrieval possible later.

📄

PDF / DOCX

PyMuPDF, pdfplumber, python-docx. Extracts text, tables. Watch for scanned PDFs (need OCR) and multi-column layouts where text order gets scrambled.

🌐

Web / HTML

BeautifulSoup, Playwright for JS-heavy sites. Strip nav/footer boilerplate. Handle pagination. Respect robots.txt. Store the URL as metadata for citations.

📊

Structured Data

CSV, JSON, SQL → convert rows to natural language strings. Consider schema-aware chunking for tables. Each row can become a Document with field metadata.

💻

Code / Markdown

Use language-aware splitters to keep function/class boundaries intact. Comment blocks often carry more retrieval signal than the code itself.

Python · LangChain from langchain_community.document_loaders import PyMuPDFLoader from langchain_community.document_loaders import DirectoryLoader # Load single PDF — preserves page numbers loader = PyMuPDFLoader("hr_policy.pdf") docs = loader.load() # Batch-load an entire folder loader = DirectoryLoader("./documents", glob="**/*.pdf") docs = loader.load() # Each doc has .page_content and .metadata print(docs[0].metadata) # → {'source': 'hr_policy.pdf', 'page': 0, 'total_pages': 12} print(f"Loaded {len(docs)} pages")
⚠️ Common pitfalls in document loading ▼
Scanned PDFs — text is an image; you need OCR (Tesseract, AWS Textract). Always check doc.page_content isn't empty after loading.

Multi-column layouts — text extraction order scrambles. Use PyMuPDF's layout mode or pdfplumber's bbox extraction.

Metadata loss — always preserve source file, page number, section heading. You'll need it for citations.

Encoding issues — UTF-8 decode errors on legacy documents. Use errors='replace' or the ftfy library.

Stage 1 — Text Extraction
Pull raw text out of binary file formats into Python strings
What happens here

File readers (PyMuPDF, pdfplumber, python-docx) convert binary formats to plain text strings. Each page or section becomes a separate Document object. The extraction step is where most information loss occurs if done carelessly.

✓ PDF text mode — reads embedded text, preserves order
✓ OCR fallback — for scanned/image PDFs (Tesseract)
✓ Table extraction — pdfplumber bbox or camelot
⚠ Multi-column PDFs — reading order often scrambled
⚠ Header/footer injection — leaks into every page
pdfplumber import pdfplumber def load_pdf(file_path): """Load PDF and extract text page by page""" documents = [] with pdfplumber.open(file_path) as pdf: for i, page in enumerate(pdf.pages): text = page.extract_text() if text: documents.append({ "content": text, "metadata": { "source": file_path, "page": i + 1 } }) return documents docs = load_pdf("company_policy.pdf") print(f"Loaded {len(docs)} pages")

Python · Full Normalization import re, unicodedata def normalize_text(text: str) -> str: # 1. Normalize unicode — curly quotes, ligatures, non-breaking spaces text = unicodedata.normalize("NFKC", text) # 2. Remove page headers / footers (common patterns) text = re.sub(r"(?im)^page\s+\d+\s+(of\s+\d+)?\s*$", "", text) text = re.sub(r"(?i)(confidential|draft|internal only)", "", text) # 3. Strip HTML tags if content was scraped text = re.sub(r"<[^>]+>", " ", text) text = re.sub(r"&[a-z]+;", " ", text) # &amp; &nbsp; etc. # 4. Collapse whitespace text = re.sub(r"\n{3,}", "\n\n", text) text = re.sub(r" {2,}", " ", text) text = re.sub(r"\t", " ", text) # 5. Fix punctuation spacing text = re.sub(r"\s([.,;:!?])", r"\1", text) return text.strip()

Stage 4 — Deduplication
Identify and remove near-duplicate documents before embedding

Duplicate content wastes embedding API budget and degrades retrieval — if the same paragraph appears in 10 documents, your top-k results will all be copies of the same text. Three approaches in increasing complexity:

Exact Hash
⚡ Cheapest

MD5 or SHA256 of the full document text. O(n) scan. Catches only byte-for-byte duplicates. Great as a first pass.

import hashlib seen = set() unique = [] for d in docs: h = hashlib.md5(d["content"].encode()).hexdigest() if h not in seen: seen.add(h); unique.append(d)
MinHash / LSH
⚖️ Balanced

Locality-sensitive hashing of shingles (n-gram sequences). Catches near-duplicates with ~1% word changes. Scales to millions of docs.

from datasketch import MinHash, MinHashLSH lsh = MinHashLSH(threshold=0.85, num_perm=128) for i, d in enumerate(docs): m = MinHash(num_perm=128) for shingle in get_shingles(d["content"]): m.update(shingle.encode()) if not lsh.query(m): lsh.insert(f"d{i}", m)
Semantic Sim.
💰 Most thorough

Embed all docs, cluster by cosine similarity > 0.95. Finds paraphrased duplicates. Best quality, but needs embedding cost upfront.

from sklearn.metrics.pairwise import cosine_similarity import numpy as np embs = np.array(embeddings) sim = cosine_similarity(embs) # Keep only docs where no prior doc has sim > 0.95 keep = [] for i in range(len(docs)): if all(sim[i][j] < 0.95 for j in keep): keep.append(i)

Python · Metadata Enrichment import os from datetime import datetime def enrich_metadata(doc: dict, file_path: str) -> dict: stat = os.stat(file_path) doc["metadata"].update({ # Source provenance "source": os.path.basename(file_path), "created_at": datetime.fromtimestamp(stat.st_ctime).isoformat(), "last_modified": datetime.fromtimestamp(stat.st_mtime).isoformat(), "file_size_kb": stat.st_size // 1024, # Org classification (rule-based) "department": classify_department(file_path), "doc_type": classify_doc_type(doc["content"]), "language": detect_language(doc["content"]), # Quality signals "word_count": len(doc["content"].split()), "quality_score": compute_quality(doc["content"]), }) return doc def classify_department(path: str) -> str: path_lower = path.lower() if "hr" in path_lower or "policy" in path_lower: return "HR" if "eng" in path_lower or "tech" in path_lower: return "Engineering" if "legal" in path_lower or "contract" in path_lower: return "Legal" return "General"

Stage 6 — Quality Gate & Validation
Catch bad documents before they pollute your vector store

Every preprocessing pipeline needs a quality gate. Documents that pass extraction but are still too short, too noisy, or in the wrong language should be quarantined and logged — not silently embedded and stored.

✓
Minimum word count
Drop docs with fewer than 20 words — usually extraction failures or nav fragments
✓
Language detection
Ensure content matches expected language; route multilingual docs to the right embedding model
✓
Encoding verification
Check for replacement characters (â€™) from mojibake — signs of double-encoding bugs
✓
Non-text ratio
If >30% of characters are non-ASCII or special symbols, it's likely an extraction failure
⚠
Log & quarantine, don't silently drop
Keep a rejected_docs.jsonl log. Your team will thank you when debugging retrieval misses
Python · Quality Gate import json, logging from langdetect import detect def quality_gate(docs: list, expected_lang="en") -> tuple: accepted, rejected = [], [] for doc in docs: text = doc["content"] words = text.split() reasons = [] # Rule 1: minimum length if len(words) < 20: reasons.append("too_short") # Rule 2: non-text ratio non_alpha = sum(1 for c in text if not c.isalpha() and not c.isspace()) if non_alpha / max(len(text), 1) > 0.3: reasons.append("high_noise") # Rule 3: language check try: lang = detect(text[:400]) if lang != expected_lang: reasons.append(f"wrong_lang:{lang}") except: pass (rejected if reasons else accepted).append( {**doc, "reject_reasons": reasons} if reasons else doc ) # Log rejections for audit with open("rejected_docs.jsonl", "a") as f: for d in rejected: f.write(json.dumps(d) + "\n") logging.info(f"Quality gate: {len(accepted)} passed, {len(rejected)} rejected") return accepted, rejected

STEP 04

Chunking

Split documents into segments so retrieval can find the right “evidence window”. Chunk size is the most impactful hyperparameter: too small loses meaning, too large lowers precision. Overlap helps when key facts span boundaries, but too much overlap wastes context and creates redundancy.

Interactive Chunking Demo

interactive_chunking_demo
Chunk size 60 chars
Overlap 15 chars
📏

Size Rules of Thumb

128–256 tokens — precise factual retrieval (great for direct Q&A)
512 tokens — balanced context window (best default for most teams)
1024+ tokens — better for multi-hop explanations, but slower and more expensive

🔁

Overlap Strategy

Use ~10–20% overlap to prevent “boundary misses” when a key sentence is split across two chunks. Too much overlap increases redundant matches and wastes prompt context window.

🌲

Recursive Splitter

Default choice. Tries separators in order: ["\n\n", "\n", ". ", " "]. This preserves paragraph → sentence → word structure, so retrieval returns chunks that read naturally. For production, prefer a token-based length function (not `len`).

Python · RecursiveCharacterTextSplitter from langchain.text_splitter import RecursiveCharacterTextSplitter splitter = RecursiveCharacterTextSplitter( chunk_size=512, chunk_overlap=64, separators=["\n\n", "\n", ". ", " ", ""], length_function=len, # swap for tiktoken for token-level splitting ) chunks = splitter.split_documents(docs) avg = sum(len(c.page_content) for c in chunks) // len(chunks) print(f"{len(chunks)} chunks created, avg {avg} chars each")
STEP 05

Embeddings

Turn text into vectors that capture meaning. Similar meanings produce similar vectors, so retrieval can match concepts even when the wording doesn’t match exactly. In practice you’ll also batch requests for speed, cache embeddings to avoid rework, and normalize vectors so cosine similarity behaves consistently.

🧮
What are embeddings? An embedding converts text into a list of numbers (a "vector"). The magic: texts with similar meaning get similar vectors. "Happy" and "Joyful" will be close together. "Happy" and "Refrigerator" will be far apart.

🔢 So we turn text into numbers that capture meaning... ▼
Why high dimensions? 1536 numbers lets the model encode many independent semantic axes simultaneously — topic, sentiment, formality, domain, entity type, and more. Each dimension captures a different facet of meaning.

Cosine similarity measures the angle between two vectors, ignoring magnitude. Two identical texts → similarity 1.0. Completely unrelated → similarity near 0. Opposites → negative similarity.

Normalization matters — always normalize embeddings to unit length before storing, so cosine similarity equals dot product (much faster to compute at scale).
STEP 06

Vector Store

A database optimized for approximate nearest-neighbor (ANN) search over dense vectors. A vector store keeps both the embeddings and their metadata, letting you retrieve fast (via ANN) while still filtering by document attributes (type, date, department, etc.) so results stay relevant.

STEP 07

Retrieval

Now the magic: given a question, embed it and search the vector store to find the most relevant chunks (usually top-k by cosine similarity). In production you often combine dense (vector) similarity with sparse (keyword) matching and optionally rerank the top-k for higher answer quality.

🔀

Hybrid Search

Combine dense vector similarity (semantic meaning) with BM25 sparse keyword matching. Reciprocal Rank Fusion merges both views, which helps when queries contain rare names, IDs, or exact phrases. Tune weighting based on what your users search for.

🏅

Reranking

After retrieving top-k with fast vector search, use a cross-encoder reranker to score (query, chunk) pairs more precisely. This often improves answer quality noticeably, at the cost of added latency—so keep k small (like 5–20).

🪄

HyDE

Hypothetical Document Embeddings: generate a plausible “answer draft” from the LLM, embed that hypothetical text, then retrieve using the hypothetical embedding. This reduces the mismatch between how users ask questions and how documents are phrased.

STEP 08

Generation

Assemble the retrieved chunks into a structured prompt and pass them to the LLM. The prompt should enforce grounding rules (answer only from provided context; cite sources) so the model becomes reliable and reviewable—rather than guessing from world knowledge.

🚀 Advanced patterns to explore next ▼
Corrective RAG (CRAG) — if retrieved docs score low, fall back to web search or broader retrieval before generating.

Self-RAG — the LLM decides whether to retrieve, critiques its own citations, and loops back to re-retrieve if needed.

Agentic RAG — wrap retrieval as a tool in a ReAct agent. Multi-step reasoning with dynamic retrieval per reasoning step.

Graph RAG — build a knowledge graph from documents. Graph traversal finds related concepts that vector similarity misses.
